# AIP SDK Development

Reference for AI agents developing the ai-projects CLI (aip). Oriented toward SDK implementation, not end-user workflows.

## Architecture

```
aip <noun> <verb> [options] [positional-args]
```

- **Entry**: `src/index.ts` — splits args into `[noun, verb, ...rest]`, routes to `commandMap[noun][verb]`, calls `command.handler(parser.parse(rest))`
- **Command map**: `src/commands/index.ts` — auto-generated by `bin/map-commands.ts`, never edit by hand
- **Commands**: `src/commands/<noun>/<verb>.ts` — one file per verb, each exports a `defineCommand(...)` result

## defineCommand

All commands use `defineCommand(config)` with a **config object**:

```ts
defineCommand({
  description: 'Shows in --help and docs/aip.md',
  options: z.object({ ... }),   // optional, defaults to z.object({})
  args: z.object({ ... }),      // optional — positional args (zod-opts .args())
  handler: async (params) => { ... },
})
```

- **options** → `--flag` style (zod-opts `.options()`)
- **args** → positional (zod-opts `.args()`). Order = schema key order. Use for required identifiers (project, task, name, type)
- **description** → First-class field, not `.describe()` on schema. Used for help and `bin/docs.ts`
- **handler params** = merged `options` + `args`

### When to Use Positional vs Options

- **Positional**: Required, primary identifiers — `project`, `task`, `name`, hook `type`
- **Options**: Optional, flags, or contextual (e.g. PWD-derived) — `--status`, `--ingest`, `--project` when optional from cwd

## Command Discovery

`bin/map-commands.ts`:

1. Globs `src/commands/*/*.ts` (excludes `index.ts`, `*.test.ts`)
2. Derives noun/verb from path: `src/commands/task/start.ts` → `task.start`
3. Generates `src/commands/index.ts` with imports and `commands` map
4. Run via `npm run build:map` (or `npm run build`)

Add command = add file. No manual registration.

## Docs Generation

`bin/docs.ts` → `docs/aip.md`:

1. Reads `commandMap` from `src/commands/index.js`
2. For each command: `command.description`, `command.options`, `command.args`
3. `extractOptions(schema)` pulls type/required/description from zod `_def`
4. Outputs markdown: description, then table with option/args (positionals labeled)
5. Run: `npm run docs`

## env.ts Conventions

- `config` from env.ts = ROOT, TEAM_HOME, NODE_ENV (env/config)
- `config` from config.ts = dirs, files, hookTypes (static)
- Paths: `TEAM_HOME/projects/<project>/tasks/<task>/`

## Hooks

Types: `pre-create`, `post-create`, `pre-update`, `post-update` (create/update only; no start/complete hooks).

- **Target**: project or task (from `getTargetDir`)
- **Pre** hooks can `exit 1` to block
- **Post** hooks run after, cannot block

## Delegation

Commands can call other commands: `task/start.ts` uses `update.handler({ project, task, status: 'in-progress' })` instead of duplicating update logic. Prefer delegation over copy-paste.

## Testing

- `src/util/defineCommand.test.ts` — defineCommand with options-only
- `npm run test` — types + unit
- `npm run cli -- <args>` — manual CLI test (builds first)
